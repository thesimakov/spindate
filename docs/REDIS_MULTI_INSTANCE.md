# Redis для нескольких инстансов приложения

Состояние живых столов, авторитетного снапшота игры и ленты событий хранится **в Redis**, если задана переменная окружения:

```bash
REDIS_URL=redis://:password@host:6379/0
```

Подойдёт любой совместимый с **ioredis** сервер (Redis OSS, Valkey, облачные Redis).

## Поведение

| `REDIS_URL` | Где хранятся данные |
|-------------|---------------------|
| **задан**   | Redis (общий для всех инстансов Next.js) |
| **не задан**| In-memory в процессе Node (как раньше; подходит только для одного инстанса) |

## Ключи (префикс `spindate:v1:`)

- `live:state` — присутствие игроков и рассадка по столам (один JSON-блоб, обновление через `WATCH`/`MULTI`).
- `authority:{tableId}` — авторитетный снапшот стола.
- `events:{tableId}` — лента событий стола.
- `profile_bundle:{userId}` — кеш ответа профиля для `/api/auth/me` (TTL 5 минут, ключ `spindate:v1:profile_bundle:{userId}`).
- `rl:chat:*`, `rl:action:*` — скользящие окна анти-флуда (`spindate:v1:rl:chat:table:{id}` и др.).

## Pub/Sub (несколько процессов WebSocket)

Каналы `spindate:v1:ws_room:{roomId}` — рассылка сообщений лобби-комнат между процессами `custom-server` ([`lib/ws-room-redis.ts`](../lib/ws-room-redis.ts)). Без этого при нескольких realtime-инстансах игроки на разных узлах не видят общий чат комнаты.

## Локальная разработка

```bash
# Пример через Docker
docker run -d --name redis -p 6379:6379 redis:7-alpine
export REDIS_URL=redis://127.0.0.1:6379/0
npm run dev
```

Без Redis приложение работает как прежде на одном процессе.

## Продакшен

- Укажите `REDIS_URL` в переменных окружения на всех репликах (один и тот же URL).
- Убедитесь, что сеть/VPC позволяет подключаться к Redis с серверов приложения.
- Рекомендуется TLS и пароль для облачных Redis (формат URL см. в документации провайдера).
